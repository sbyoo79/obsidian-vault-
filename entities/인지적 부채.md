# 인지적 부채

AI 가 생성한 코드를 **이해 없이 받아들였을 때 누적되는 부채**. [[백명석]] 이 [[brunch-cleancode-ai-no-silver-bullet]] (2026-06-08) 에서 도입한 framing.

## 개요

기술 부채(technical debt) 와 구분되는 개념:

- **기술 부채** = 코드 품질·구조의 미래 비용. 리팩토링·재설계로 해소 가능
- **인지적 부채** = 코드를 만든 사람조차 **왜 그렇게 됐는지 설명 못 하는 상태**. 문제 발생 시 **대처 자체가 불가능**

핵심 차이: 기술 부채는 코드에 산다, 인지적 부채는 **이론(theory)의 부재** 에 산다 — [[Programming as Theory Building]] 의 직접 연장.

## 본 vault 에서의 의미

AI 시대 시니어 vs 주니어 격차의 본질을 한 단어로 압축한 framing. [[Vibe Coding]] 의 위험을 "이해 없이 받아들이는 단계가 인지적 부채로 직결" 로 재정의 — 단순한 코드 품질 비판이 아니라 **운영 가능성의 부재** 라는 더 강한 진단.

본인 [[Agentic Engineering]] 워크플로 (Codex 구현 → Claude 리뷰 → 본인 docs) 의 docs 단계가 사실상 인지적 부채 방지 메커니즘. [[Eval]] 도 같은 결.

## 핵심 원리

1. **부채의 단위가 코드가 아닌 이해.** 누적되어도 즉시 드러나지 않음 (정상 동작 시)
2. **결재 시점 = 장애·요구 변경.** 코드를 고쳐야 할 때 비로소 "왜 이렇게 됐는지" 모름이 노출
3. **조직 단위에서도 발생.** 베테랑의 트라이벌 지식(tribal knowledge) 이탈로 같은 패턴이 회사 차원에서 발생 → AWS 2025-10 장애 사례
4. **방지 수단:** 명세(spec) 의 deterministic 화, 검증을 통한 멘탈 모델 구축, 스크래치 리팩토링, AI 에게 "내가 이해할 수 있는 수준" 설명 요청
