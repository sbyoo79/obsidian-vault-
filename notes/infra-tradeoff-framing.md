# 인프라 trade-off framing

본인의 인프라·운영 의사결정 사상 living doc. 새 인프라 도구·패러다임 (Docker, K8s, Terraform, Prometheus, CI/CD 통합 등) 도입 검토 시 다시 펼쳐서 일관성 있게 결정.

> CD가 지금 빈약한게 사실이긴해. 거기다 난 인프라쪽 지식이 부족한지라 멋지게 랩핑 하기도 뭐하고... 거기다 구성원들 교육 비용도 발생해서 현상 유지 중이긴 하다. 변경을 하면 지속적으로 관련 내용 운영 및 유지보수도 해줘야 안정적인데 그렇게 해줄 dev-ops 분들도 없고... 트레이드 오프로 따지면 비용이 더 들어갈꺼 같아서 일부로 CD쪽은 크게 건드리지는 않는 상황이야. — 본인 발화 (2026-05-22)

## 운영 원칙

본 노트의 운영 원칙은 [[ai-overwhelm-framing]] 의 짝:
1. **새 인프라 도구 도입 검토 시 항상 여기 와서 framework 적용**
2. **현상 유지 결정도 명시적 의사결정** — 게으름이 아닌 trade-off 분석 결과
3. **새 framework 변수가 추가될 때만 갱신** — 그 외엔 안 건드림
4. **결정 사례를 누적** — 같은 사상의 일관된 적용 증거
5. **회피 ≠ 영구 거부** — "여유 있으면 검토" 가 본인 stance 의 정확한 결. 현 결정은 **현재 조건 (시간 / 인력 / 인지 부하) 에서의 결정**. 조건이 변하면 재평가 (재검토 트리거 참조)

## 현재 인프라 상태 (2026-05 기준)

- **컨테이너**: Docker 미사용
- **IaC**: Terraform 미사용
- **기동 방식**: nohup 으로 직접 기동
- **CD**: 빈약 — Jenkins 빌드 후 SSH + 원격 sh 실행이 전부
- **CI**: Jenkins (회사 표준)
- **모니터링·로깅·관측 도구**: 별도 명시 없음 (필요 시 본인이 점검)
- **인력**: 전담 dev-ops 부재. 본인이 dev + dev-ops 일부 겸직

## 본인 자기 인식

- 인프라 측 지식 부족이라는 정직한 자기 평가
- "멋지게 랩핑 하기도 뭐하고" — 미숙한 상태에서 도입 시 오히려 안정성 ↓ 위험 인식
- 본인 강점은 백엔드 개발·도메인 설계·복잡 마이그레이션. 인프라 자동화는 강점 영역 X

이 자기 인식은 [[self-profile]] 의 약점 영역과 일치. **단, 약점 보완을 위한 학습 영역이라기보다 의도적 회피 영역**.

## Trade-off framework — 새 인프라 도입 검토 시 5 변수

새 인프라 도구 / 패러다임 도입 검토 시 다음 5 변수를 가중치 매겨 평가:

| 변수 | 질문 | 본인 컨텍스트 기본값 |
|------|------|--------------------|
| **도입 비용** | 학습 + 셋업 + 마이그레이션에 얼마 걸리나? | 본인이 직접 → 시간 부족 |
| **교육 비용** | 팀원들이 적응하는 데 얼마? 운영 단순도 ↓ 면 부담 ↑ | 부담 큼 |
| **유지보수 인력** | 도입 후 운영할 dev-ops 가 있나? | 부재 — 본인이 다 떠안음 |
| **운영 안정성** | 현재 안정성 → 도입 후 안정성. 일시 하락 vs 장기 향상 | 현재가 안정 |
| **명확한 이득** | 도입으로 얻는 게 구체적·즉시 측정 가능한가? | 불명확하면 도입 X |

**의사결정 규칙**: 5 변수 중 "명확한 이득" 이 다른 4 변수의 총합을 넘을 때만 도입. 애매하면 **현상 유지**.

### 실측 사례 — "전문 지식 부담" 변수의 무게

본인 관찰: **K8s 운영** = 전문 지식 부담이 만만치 않다는 게 다른 운영 사례에서 확인됨. 도입 후의 학습·운영 깊이가 도입 결정 시 가늠한 것보다 훨씬 큰 패턴. 이는 framework 의 "도입 비용 + 교육 비용 + 유지보수 인력" 변수들이 추상이 아닌 **실측 무게** 를 가진다는 본인 근거.

→ 새 인프라 도구 검토 시 본인은 이 변수들을 **공식 문서·홍보 자료** 가 아닌 **실제 운영하는 회사·팀의 부담 관찰** 기준으로 평가.

## 재검토 트리거

다음 조건이 충족될 때 본 stance 자체를 재검토:

- **전담 dev-ops 인력 합류** — 유지보수 부담 해소
- **운영 안정성에 큰 균열** — 현 nohup 기동·SSH 배포가 사고·장애를 만들기 시작
- **본인 인프라 학습 의지·여유 발생** — 약점 영역을 강점으로 전환 결정. **"여유" = 시간 + 인지 부하 여유** 모두 충족 필요 (현재는 이직 준비 + 회사 일 + AI 도구 운영으로 인지 부하 ↑ 상태)
- **회사 규모 확대로 수동 운영의 한계 도달** — 자동화 없으면 운영 불가능한 시점
- **외부 도구가 학습·운영 부담을 크게 낮춰주는 옵션 제공** — 예: 매니지드 서비스, 풀스택 통합
