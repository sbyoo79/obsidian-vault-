## 약점 / 학습 필요 / 리스크

### Docker / K8s — 본인 자기 평가 (2026-05-26 결정)

**본인 발화 (그대로)**:
> 회사가 필요로 하면 공부 해야지. 대충 어떤 물건인지는 아는데 운영은 안해봤어.

이는 [[infra-tradeoff-framing]] 운영 원칙 5 ("회피 ≠ 영구 거부, 조건부 결정") 의 정확한 적용:
- "회사 필요" 라는 조건이 충족되면 학습 진입 — 본인의 명확한 학습 의지 표명
- K8s 가 [[infra-tradeoff-framing]] 의 "공식 자료 X, 실제 운영 부담 관찰 O" 인식론으로 형성된 회피 영역이었으나, 회사 환경이라는 새 조건이 회피 해제 트리거

**자기 평가 정확도**:
- 개념 이해: ✅ ("대충 어떤 물건인지는 아는데")
- 본격 운영 경험: ✗ ("운영은 안해봤어")
- 학습 의지: ✅ 조건부

**면접 / 이력서 표현 안**:

이력서 (기술 스택 절):
> Docker / Kubernetes — 개념 이해, 본격 운영 경험 제한적. 회사 환경 적응 시 학습 의지 충분 (21 년차 시니어 학습 속도)

면접 답변:
> K8s 가 어떤 물건인지는 압니다. 다만 본격 운영은 안 해봤습니다. 회사가 필요로 하면 공부할 의향이 있고, 21 년차 백엔드라 학습 속도는 빠른 편입니다. 본인이 의도적으로 회피해 온 영역이긴 했는데, 그건 현 회사 환경에서 도입 비용 / 교육 비용 / 유지보수 인력 trade-off 가 안 맞았기 때문이지 학습을 포기한 게 아닙니다. 회사가 이미 운영 중이라면 학습 환경이 갖춰져 있는 것이니 빠르게 적응 가능합니다.

**왜 이 답이 강점이 되는가**:
1. **거짓말 안 하는 시니어** — "운영해 봤습니다" 가 아닌 정직한 자기 평가
2. **컨텍스트 인식하는 시니어** — 단순 "안 해봤다" 가 아닌 본인이 그 결정에 도달한 trade-off 까지 설명 가능
3. **학습 의지 명확** — "회사 필요" 라는 조건만 충족되면 진입
4. **회사 입장**: 인프라 본격 운영자 ≠ 백엔드 팀장 우선 역량. 팀장 역할은 도메인·아키텍처·팀 리딩이 핵심, K8s 는 합류 후 학습 가능
