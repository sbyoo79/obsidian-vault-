### 기본 톤 — 마스터 기준 (에이블리 v4 패턴)

다음 매칭부터 기본값으로 적용:

- **헤더 구조**: 격식 번호 매김 X. 자연스러운 절 제목 (한 줄 자기 소개 → 개인 신상 → 학력 → 근무 이력 → 핵심 보유 역량 → 기술 스택 → 경력 세부 → 지원 동기)
- **핵심 보유 역량**: 9 sub-section 단락 서술식 (Commerce / 팀 리딩 / Java·Spring / MSA·DB / 마이그레이션 / 0→1·종단 / OOP·TDD / AI-aug / 언어 유연성). 한 줄 자기 소개에서 회사 향 강조점 조정
- **기술 스택**: C 안 (진화 흐름 + Self-Assessment 표) — 상단 통합, 각 프로젝트 기술 줄 제거
- **경력 세부**: 단락 서술식 + 리드급 디테일 sub-section (리드플레닛·티몬·케이타운포유·MAXVISOR 4 곳) + 각 절 끝 "이 시기에 형성된 역량" 한 줄
- **지원 동기**: 회사 향 단락 (2~3 단락, bullet 최소)
- **회피**: ✅ / ✨ / "차별 강점 1 호" 격 표현 / "본 후보 중 최강" 같은 마케팅 어휘 / 매칭 점수 메타 표기 / 면접 핵심 문장 메타 절

### 예외 톤 — 격식 서식화 (휴넷 v5 패턴)

다음 조건에서만 적용:
- **회사 도메인**: EdTech / SI 대기업 / 전통 산업 / 격식 문화 명확한 곳
- **외부 신호**: 추천인·HR·면접관 측에서 "서식화" / "정형화" 피드백
- **자기소개서 분리 요구**: 한국 격식 이력서 + 자기소개서 분리 양식 명시

격식 적용 시:
- 번호 매김 1~7 큰 절 (인적·학력·경력·핵심·기술·상세·자기소개서)
- 인적 사항 표 확장 (생년월일 / E-mail / 지원 포지션)
- 경력 사항 요약 표
- 자기소개서 sub-section 분리 (적합성 / AI 도구 / 부족 영역 인식)
- 핵심 역량과 경력 세부는 **마스터 톤 단락 서술 유지** (격식 + 스토리 하이브리드)

- **영어**: 국내 한정이면 큰 문제 아니지만 시장 전체 중요도가 올라가는 중. 앞으로 더 중요해질 거라고 보고 vault 의 영문 파일명 유지, 영문 source ingest 등 일상 노출을 늘리는 중.
- **외부 발표 / 외부 활동**: 21 년 경력 대비 외부 컨퍼런스·오픈소스·기술 블로그 적음. 짬짬이 스터디 + **사내 세미나 발표** 로 발표 능력 향상 중. 외부 활동도 차차 늘릴 계획.

### "가장 자랑스러운 프로젝트?"
- (후보) **싸이월드 파일시스템 컨버전** — 3.2 PB / 320 만 건 / 윈도우 스토리지 300 대 변환. 규모·위험 양쪽 다 큰 작업
- (후보) **MAXVISOR 0→1 + 리드플레닛 종단 개선** — 동일 도메인의 설계자가 7~9 년 후 자기 시스템을 직접 책임지고 개선
- (후보) **리드플레닛 메모리 절감** — 256~512 GB / 90%+ → 128 GB / 10~15%, 200 인스턴스 → 1

### "실패 경험?"
- **티몬 브랜드 홈**: 기획 단계부터 무리, 실적 부진 (브랜드 5 개 stop). 본인 의구심이 있었는데 더 강하게 신호 보내지 못한 게 학습 포인트
- **티몬 구독 서비스**: PMF 부족, 구독자 100 명 미만. 엔지니어가 잘 만들어도 비즈니스 결과는 별개

### "기술적 견해" 류
- **Spring Batch 사장 견해**: "대체할 수 있는 게 많아서 사장되어야 할 물건" 으로 본다. 대안: Airflow, Spring Cloud Task, MQ + worker 구조, 자체 Python 파싱 MS 등. 실제로 리드플레닛에서 대량 파일 처리를 Python MS + MQ + Worker 로 분리한 사례 직접 있음 — 면접 시 구체 사례로 풀어낼 수 있음.

## 프로젝트별 솔직한 자기 평가 (master 에는 안 들어감)

### 티몬 — 브랜드 홈 (2022.05 ~ 2022.08)
- 결과: 기획 단계부터 무리한 진행, 실적 부진
- 트래픽: 의미 있는 수치 미달
- 브랜드 수: 5 개 정도에서 stop
- 학습: 기획 검증이 부족한 프로젝트의 리스크를 직접 경험. "출시" 만 잘 되어도 비즈니스 결과는 별개라는 것
- 면접 활용: "실패 경험" 질문에서 사용 가능 — 본인이 미리 의구심 표현했는지, 어떤 신호를 놓쳤는지, 다음에 어떻게 다를지

### 티몬 — 구독 서비스 (2021.10 ~ 2022.03)
- 결과: 구독자 100 명도 안 됨
- 자평: "하지 말았어야 했던 프로젝트"
- 학습: PMF 없는 사이드 프로덕트는 엔지니어가 잘 만들어도 망함
- 면접 활용: 같은 질문군에서 사용 가능

### 티몬 — 공통 영역 (딜 상세 / GNB / 카테고리)
- PV / 트래픽: 점점 감소 추세 (큐텐 인수 후 전사 트래픽 하락)
- 정확한 수치 기억 X
- master 표기: "운영·안정성 측면 책임" 으로 표현, 트래픽 정량은 빼는 게 안전

### 티몬 — 자동화 매장 ES 증분 색인 (2023.04)
- 1 억 건 / 3 시간 색인 → 강한 임팩트, master 에 정량 포함
- 증분 처리량은 정확 기억 X — 면접에서 물어보면 "정확 수치는 기억나지 않으나 전체 재색인 대비 X% 수준" 같은 표현으로

### 티몬 — 상품 DB 이전 + SKU (2020.10 ~ 진행 중)
- 500 만 건 정리 — 강한 임팩트, master 에 포함
- SKU 호출량 미상

### MAXVISOR (2016.11 ~ 2019.07) — 솔직한 자기 평가
- "멍청한 시절이어서 레이어드 아키텍처를 잘못 사용" — 본인 표현
- 무에서 유를 창조한 의미는 있음. 다만 초기 아키텍처 선택이 후속에서 부채로 작용
- 면접 활용: "본인의 약점" / "성장 경험" 질문 — 시니어다운 자기 인식 답변 소재
- 답변 초안: "스타트업에서 0→1 로 백엔드를 설계할 때 빠르게 만드는 데 집중해서 레이어드 아키텍처의 분리 원칙을 일부 잘못 적용한 결정들이 있었습니다. 그 결과가 시간이 지나 부채가 되는 것을 같은 회사의 후신인 리드플레닛에서 직접 확인했고, 지금은 그 부채를 본인이 책임지고 VSA 로 점진 전환 중입니다. 시간 차를 두고 자기 결정을 다시 보는 경험이 시니어로서의 설계 직관을 가장 많이 키워준 시기였습니다."
