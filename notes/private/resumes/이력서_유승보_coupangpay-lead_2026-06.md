## 4. 핵심 보유 역량

### 4.1 팀 리딩과 기술 의사결정

개발팀장·BackOffice 스쿼드 리더 등 조직을 리딩하며 업무 배분, 프로젝트 리딩, 구성원·기술 멘토링, 기술 의사결정을 담당해 온 경험이 누적되어 있다.

- 티몬: 서비스개발 3 팀 팀장 (최대 13 명) — 1-on-1·멘토링·전사 프로젝트 사전 조율
- 케이타운포유: 기능 조직 외 목적 조직으로 신설된 BackOffice 스쿼드 리더 (백엔드 4 명)
- 리드플레닛: 오피스개발팀 팀장 — 작업 분배, 코드 리뷰, 멤버 멘토링, 기술 의사결정

### 4.2 결제·정산 인접 경험과 데이터 정합성

쿠팡페이의 결제 플랫폼·정산 영역에 직접 맞물리는 인접 경험이 누적되어 있다. 결제 시스템 직접 운영 경험은 없으나, 외부 결제·금융 시스템 연동과 매출·정산 데이터의 대사·정합성 작업은 실무로 다뤘다.

- **외부 결제·금융 시스템 연동**: MAXVISOR 에서 POS (KICC / 스마트로 / 메타시티) 매출 데이터 연동과 키움증권 가상계좌 연동을 설계·구현. 외부 시스템 응답 지연·중복·실패 시나리오를 미리 검토하고 보상 처리를 함께 구현
- **DB 대사·정합성 보정**: 케이타운포유 BackOffice 스쿼드에서 이벤트 데이터 등록·주문 오입력 수정 등 **수백만 건 규모** 의 데이터 정합성 보정 작업, 티몬 상품 DB 약 **500 만 건** 이관 + SKU 단위 구조 개선
- **메시지 정합성 패턴**: 리드플레닛에서 Outbox Pattern 기반 메시지 전송 구조 도입 — DB 트랜잭션과 메시지 발행을 동시 commit, 데이터 유실 방지와 재처리·상태 추적 가능

### 4.3 공통 기반 기술 영역의 운영 경험

전사 공유 기반 기술 — 인증, 알림, API 통합, 데이터 통합 — 영역에서 실제 운영 경험이 누적되어 있다.

- **인증·인가**: 인플루엔셜 (Welaaa) 에서 삼성 빅스비 OAuth 2.0 회원 연동 구현, MAXVISOR 에서 Spring Security 기반 인증 시스템을 처음부터 설계·구축
- **알림·이벤트 릴레이**: 리드플레닛에서 Outbox Pattern 기반 메시지 전송 구조를 도입해 데이터 유실 방지와 메시지 재처리·상태 추적이 가능한 구조 확보
- **데이터 통합**: 리드플레닛에서 3 자 물류·POS·자체 수발주 매출 등 파편화된 데이터를 한 곳으로 모으는 Data Warehouse 를 구축, Gold Layer 까지의 집계와 보고서 산출 연동 완료

### 4.4 Java / Spring 기반 시스템 설계와 운영

Spring 2.x 부터 Boot 2.7 까지 직접 운영하며, 단순 사용을 넘어 시스템 아키텍처와 데이터 모델링을 주도해 왔다.

- MAXVISOR 에서 Spring Boot + JPA + Spring Batch + Spring Security + ELK 기반 백엔드를 처음부터 설계·구현
- 리드플레닛에서 자바 1.8 + Spring Boot 1.4.2 환경을 자바 17 + Spring Boot 2.7.18 까지 안전하게 전환
- 잘못 나눠진 레이어드 구조는 도메인별 Vertical Slice Architecture 로 점진 전환 중

### 4.5 MSA 환경 운영과 RDB 설계·튜닝

- 리드플레닛에서 레거시의 대량 파일 처리 로직을 Python 파싱 마이크로서비스 + MQ + Worker 구조로 분리해 메인 시스템 부하 감소와 장애 시 재처리 가능 구조를 확보
- 리드플레닛에서 멀티테넌트 환경의 DB 커넥션과 배치 병목 이슈를 개선
- 카탈로그·주문·정산·결제 등 다양한 도메인의 데이터 모델 설계·튜닝 경험 누적
- 티몬에서 자동화 매장 상품 약 1 억 건에 대한 ES 증분 색인을 도입, 상품 DB 약 500 만 건을 이전하면서 SKU 단위 구조 개선 진행

### 4.6 직접 설계·구축한 시스템의 종단 책임

MAXVISOR (2016~2019) 에서 백엔드 기술 스펙 정의와 구조 설계를 직접 주도해 제로 베이스에서 운영 가능한 수준까지 구축한 ERP SaaS 를, 7~9 년 후 후속 회사 리드플레닛에서 같은 인물이 한계를 인지하고 개선하고 있다. 당시 설계 결정 일부가 시간이 지나 부채로 작용했고, 그것을 회피하지 않고 직접 회수하는 사이클을 지나는 중이다. 자기 설계를 시간 차를 두고 다시 보는 보기 드문 경험이 시니어로서의 설계 직관을 단단하게 만들었다.

### 4.7 위험·복잡 마이그레이션 (대규모 운영 환경)

- 싸이월드 파일시스템 윈도우 → 리눅스 컨버전 (약 3.2 PB 규모 데이터, 약 320 만 건 메타, 윈도우 스토리지 약 300 대) 을 운영 환경에서 안전하게 완료
- 자바·Spring Boot 메이저 버전 업그레이드 진행 (자바 1.8 + Spring Boot 1.4.2 → 자바 17 + Spring Boot 2.7.18)
- 다중 어플리케이션 구동 구조를 단일 어플리케이션 + 멀티테넌트 아키텍처로 전환해 메모리 256~512 GB / 90 %+ 점유 환경을 128 GB / 10~15 % 점유 환경으로 축소 (메모리 비용 대폭 절감)

### 4.8 객체지향 설계와 테스트 주도 개발

객체지향 설계 원칙과 테스트 주도 개발의 가치에 대한 이해를 바탕으로 업무를 수행한다.

- 레이어드 → VSA 점진 전환과 함께 단위 테스트·인수 테스트 코드를 지속 추가하며 변경 영향 범위를 줄이는 방향으로 작업
- 주 사용 아키텍처는 VSA 이며, 레이어드 → VSA 전환과 함께 Spring Modulith 적용도 병행 중 (도메인별 역할·책임 분리 및 캡슐화)

### 4.9 AI-augmented engineering 적용 (2025 ~)

AI 코딩 도구를 단일 도구가 아닌 멀티 에이전트 워크플로로 운영 중이다.

- 기초 설계와 문서화는 GPT, 구현은 Codex, 단위 리뷰는 Claude 로 역할을 분리해 단일 LLM 의존을 피하고 상호 검증 효과를 확보
- 회사 단위로는 공통 작업 표준 문서 (AGENTS.md) 를 작성해 선행 분석 → 티켓 생성 → 검증 루프 → 문서화의 작업 흐름을 정착
- 광고 플랫폼과 Data Warehouse 같은 신규 시스템을 각 약 1 주 내 구현·검증 완료하면서 이후 인계 가능한 수준의 문서화도 함께 진행 (단순 vibe coding 이 아닌 검증 루프·컨텍스트 엔지니어링·문서화 동반)

## 6. 상세 경력

### 6.1 리드플레닛 (2024.12 ~ 현재) / 오피스개발팀 / 팀장 / 백엔드 개발

중·소규모 프랜차이즈 업계 전용 B2B ERP SaaS 의 백엔드 개발팀을 맡고 있다. 팀 운영과 시스템 책임을 병행하는 자리이며, 본인이 2016 ~ 2019 MAXVISOR 시기에 처음부터 설계·구현한 시스템의 후신을 같은 인물이 돌아와 개선 중인 사례이다. 동시에 회사 단위의 AI 코딩 워크플로 표준을 정착시키는 작업을 병행하고 있다.

#### 팀 운영과 기술 의사결정

백엔드 개발팀의 작업 분배, 코드 리뷰, 멤버 멘토링, 기술 의사결정이 팀 운영 측 책임이다.

- **작업 분배**: 팀원별 도메인 익숙도와 학습 의지를 매칭해 일감을 나누되, 신규 시스템 (광고 플랫폼·DW 같은 신규 영역) 은 본인이 직접 첫 진입해 패턴·구조를 잡고 이후 인계하는 방식으로 운영한다. 익숙한 도메인은 팀원 주도 + 본인 리뷰, 새 영역은 본인 주도 + 팀원 공동 작업 + 문서화 인계.
- **코드 리뷰**: 휴먼 리뷰 외에 AI 리뷰 (Codex 구현 → Claude 리뷰, 그 역) 를 cross-check 매트릭스로 운영해 단일 시점·단일 리뷰어의 사각지대를 줄이고 있다.
- **멤버 멘토링과 1-on-1**: 정기 1-on-1 으로 작업 진행 외에 커리어 방향·동기·블로커를 함께 다룬다. 회사 변동기 (정책·인력 변화 등) 에는 본인이 회사 측 결정의 기술적 의미를 번역해 팀원의 불확실성을 줄이는 역할도 맡는다.
- **기술 의사결정**: 아키텍처 변경 (단일 어플리케이션 + 멀티테넌트 전환, 레이어드 → VSA) 같은 큰 결정은 본인이 분석·계획 문서로 트레이드오프를 명시한 뒤 팀·관리자와 사전 협의를 거쳐 진행한다. 검증 가능한 단계로 쪼개 단계별 회고와 함께 진행하는 것이 원칙이다.
- **AGENTS.md 표준 운영**: 회사 공통 AGENTS.md (AI 코딩 워크플로 표준 — 선행 분석 → JIRA 티켓 → 브랜치 → 본 작업 → 검증 루프 → 커밋 → Jira 전송) 를 직접 작성·운영 중이며, 단순 작성에 그치지 않고 팀이 실제 적용하며 발견된 빈틈을 회고로 반영해 표준을 진화시키는 구조로 가져가고 있다.

#### 시스템 책임 — 멀티테넌트 ERP SaaS 운영과 개선

다수 테넌트를 사용하는 멀티테넌트 백엔드 시스템 설계·운영, 매출·정산·물류 데이터의 집계·검증·보정 프로세스, 내부 관리자 (Back-office) 시스템 API 와 배치 로직, 대량 엑셀 업로드 기반 업무 프로세스 안정화가 시스템 측의 주요 책임이다. 핵심 성과로는 다음이 있다.

- **MSA 분리로 메인 시스템 부하 해소**: 레거시의 대량 파일 처리 로직을 Python 파싱 마이크로서비스 + MQ + Worker 구조로 분리해 메인 시스템 부하를 줄이고 장애 발생 시 재처리 가능 구조를 확보했다.
- **Outbox Pattern 도입으로 데이터 정합성 확보**: 메시지 전송 구조에 Outbox 패턴을 도입해 데이터 유실 방지와 메시지 재처리·상태 추적이 가능하도록 했다. 결제·정산 영역의 부분 실패 회피 요구와 동형 패턴.
- **멀티테넌트 환경의 병목 개선**: DB 커넥션과 배치 병목 이슈를 진단·개선해 운영 안정성을 회복시켰다.

#### 신규 시스템 — AI 도구 기반 단기 검증 사이클

광고 플랫폼과 Data Warehouse 두 신규 시스템을 2026.03 에 각 약 1 주 내 구현·검증 완료했다. 두 프로젝트 모두 멀티 에이전트 워크플로 (GPT 설계·문서 / Codex 구현 / Claude 단위 리뷰) 만으로 진행했고, 이후 다른 작업자가 이어갈 수 있는 수준의 문서화도 함께 마쳤다.

- **광고 플랫폼 (LeadUS 배너 광고)**: 가맹점 수발주 프로그램 LeadUS 에 배너 광고를 추가하는 프로젝트로, 기본 기능의 광고 플랫폼을 일정 내 구축 완료했다. Impression 과 Click 이 정상 적재되어 운영 중 일 평균 12,000 ~ 15,000 impression 수준으로 집계되고 있다. 기술 스택은 MySQL · Java 21 · Spring Boot.
- **Data Warehouse**: 3 자 물류·POS·자체 수발주 매출 등 파편화되어 있던 매출 데이터를 한 곳으로 모으는 DW 를 구축했다. 본사의 추이·위험 가맹점 분석에 활용하고, 추후 AI 연동으로 가맹 본사 업무에 데이터를 제공하는 것이 목표이다. 채널별 리팩토링과 DW 자체 설계·구현을 담당했고, POS 와 물류 데이터의 Gold Layer 까지 집계 + Claude 연동으로 보고서 산출까지 테스트를 완료했다. 기술 스택은 MySQL · Python · FastAPI.

#### 종단 책임 — Legacy Office 개선 및 아키텍처 변경 (2025.01 ~ 계속)

본인이 2016 ~ 2019 MAXVISOR 시기에 설계·구현한 ERP SaaS 의 후신이다. 초기 구축이 테넌트별 어플리케이션 분리 구조였고 자바 1.8 + Spring Boot 1.4.2 환경이라 기술 부채가 누적되었는데, 그 결과를 같은 인물이 돌아와 풀어가는 사이클이다. 대상 코드베이스는 maxvisor-api 기준 Java 파일 약 2,000 개, 약 25 만 LOC 규모이다.

본인이 주도한 핵심 결정과 결과:

| 영역 | 변경 전 | 변경 후 | 의미 |
|------|--------|--------|------|
| 인스턴스 아키텍처 | 테넌트별 어플리케이션 분리 (약 200 개 인스턴스) | 단일 어플리케이션 + 멀티테넌트 | 인스턴스 수 1 / 200 로 축소, 운영 단순화 |
| 메모리 점유 | 256 ~ 512 GB 환경의 90 %+ | 128 GB 환경의 10 ~ 15 % | 서버 사양·인스턴스 수 동시 축소로 인프라 비용 큰 폭 절감 |
| 파일 저장소 | 자체 파일 서버 (저장 공간 압박) | S3 이전 | 운영 부담·비용·확장성 동시 개선 |
| 도메인 구조 | 잘못 나눠진 레이어드 | Vertical Slice Architecture 점진 전환 | 도메인별 응집도 향상, 향후 변경 영향 범위 축소 |

운영 측면에서는 분석·계획을 프로젝트 내 문서화해 이후 작업자가 이어가도 무리 없는 수준의 인계 가능 상태를 유지하고 있다. 쿼리 복잡도 축소와 가독성 개선, 단위·인수 테스트 코드 추가, 개선 작업과 신규 기능 개발 병행을 지속 중이다.

**이 시기에 형성된 역량**: 팀 운영과 시스템 책임 병행, MAXVISOR 후신 종단 책임 (전신 시스템 설계자가 7 ~ 9 년 후 본인 부채를 직접 풀어가는 사이클), 멀티테넌트 ERP SaaS 아키텍처 변경 + 인프라 비용 절감, MSA 분리·Outbox 패턴 등 데이터 정합성·운영 안정성 패턴 도입, AI 코딩 워크플로 회사 단위 표준화 (AGENTS.md), 단기 검증 사이클 (광고 플랫폼·DW 각 1 주) 운영.

### 6.2 케이타운포유 (2023.07 ~ 2024.12) / 백엔드개발팀 / 매니저 / 백엔드 개발

K-Pop / 한류 관련 글로벌 commerce 플랫폼에서 BackOffice 스쿼드 리더 (백엔드 4 명) 를 맡았다. 기능 조직 외 목적 조직으로 신설된 스쿼드의 리더로서 실무 진행과 업무 분배를 병행하고, 리더 회의와 타 부서 소통 창구 역할을 담당했다. 케이타운포유 서비스 운영 측면에서는 BackOffice 신규 기능 개발, 서비스 영역 운영상 추가 기능 개발과 CS 대응, 그리고 수백만 건 규모의 데이터 대량 처리 (이벤트 데이터 등록, 주문 오입력 수정 등) 가 주요 작업.

#### 스쿼드 리더 역할의 실제 운영 방식

- **목적 조직 신설기의 경계 설정**: BO 스쿼드는 기능 조직과 별개로 신설된 목적 조직이었기에, 기존 기능 조직 (서비스팀·인프라팀 등) 과의 우선순위 충돌이 잦았다. BO 스쿼드 입장에서 요구사항을 사전 정리해 타 부서와의 협의 비용을 줄이고, 팀 내부에는 변동성 큰 요구를 흡수할 수 있는 작업 큐 구조를 두었다.
- **4 명 운영 — 실무와 리딩 50:50**: 작은 팀에서는 리더가 실무를 놓으면 균형이 깨지기 쉽다고 본다. 어려운 영역 (대량 데이터 보정·수백만 건 규모의 주문 오입력 수정 등) 은 본인이 먼저 한 사이클을 직접 돌려 본 뒤 팀원과 함께 보면서 넘겼고, 익숙한 영역은 팀원 주도 + 본인 리뷰로 운영.
- **상품 (SKU) 구조 개선의 의사결정 참여**: 신규 SKU 분리 설계 단계에 BO 스쿼드 리더로 참여해 AS-IS 분석과 개선 방향을 제시했다. SKU 와 전시 상품의 분리, BO 운영자 관점에서의 사용성, 데이터 정합성 보장 메커니즘 등을 함께 협의했다.

#### 프로젝트 — 상품 구조 개선 (2024.04 ~ 본인 재직 종료 시점)

상품을 물류 SKU 와 전시 상품으로 분리하고 SKU 를 신규 개발하는 작업. 상품 관리 영역의 추가 개발도 동반되었다. 본인은 기초 설계 (SKU) 참여, AS-IS 상품 구조 분석 후 개선 방향 제시, 실행 계획 수립과 업무 분배, Backend 개발을 담당했다.

**이 시기에 형성된 역량**: 기능 조직 외 목적 조직 스쿼드 리더십, K-Pop / 한류 글로벌 commerce 도메인의 BackOffice 운영, 수백만 건 규모 데이터 보정·정합성 작업 (주문·이벤트 데이터 정합성), 상품 (SKU) 도메인 구조 개선 설계.

### 6.3 티몬 (2020.08 ~ 2023.06) / 서비스개발 3 팀 / 팀장 / 백엔드 개발

큐텐 인수 후 인력 이탈로 다중 범위를 커버하는 시기에 합류해 티몬의 서비스 영역 전반 (메인 홈·추천 영역, 기획전, 카테고리, 자동화 매장 등) 운영을 담당했다. 서비스 영역 API 와 어드민 개발, 지속적 개선·안정화, 신규 기능 추가, 그리고 팀 내 다양한 영역을 잇는 cross-functional 역할이 주요 책임.

큐텐 인수 후 시니어 개발자에서 팀장으로 전환되어 최대 13 명 규모의 팀을 운영했고, CTO 부임 후 상품 상세 영역이 별도 팀으로 분리되면서 8 명 규모로 조정되었다. 팀 내 업무·일정 조율과 팀 운영, 전사 프로젝트 사전 조율과 기술 협의, 팀원 기술 지원과 멘토링, 주기적 1-on-1 을 통한 팀원 상태 점검과 멘탈 케어가 팀장 역할의 주요 내용. 동시에 티몬 공통 영역 (딜 상세, GNB, 카테고리) 의 Back-End / Batch / 외부 연동을 담당하고, PHP 기반 마이티몬 등 레거시 영역을 포함한 전 스택을 책임졌다.

#### 리드 역할의 실제 운영 방식

- **인력 이탈기의 다중 범위 커버**: 큐텐 인수 후 인력이 빠지면서 잔존 인원으로 메인·추천·기획전·카테고리·자동화 매장 등 서로 다른 도메인을 동시에 운영해야 했다. 도메인별 owner 를 지정해 책임 영역을 명확히 하면서도, 도메인 간 의존성이 큰 변경은 본인이 통합 책임자로 들어가 의사결정 시점을 통제했다. 다중 범위는 한동안 본인이 안고 가다가, 안정화된 영역부터 팀원에게 단계적으로 넘겼다.
- **업무 분배 방식**: 팀원 13 명을 도메인 익숙도와 학습 의지에 따라 매칭하되, 큐텐 인수 같은 격변기에는 단기 운영 안정성과 중장기 멤버 성장을 함께 고려해 일감을 재분배했다. CTO 부임 후 8 명으로 조정되고 상품 상세가 별도 팀으로 분리될 때도 인계 범위와 책임 경계를 사전 협의했다.
- **멘탈 케어와 번역 역할**: 격변기에는 정기 1-on-1 빈도를 더 자주 가져갔고, 회사 측 정책 변화 (인수·조직 개편 등) 의 기술적·운영적 의미를 본인이 번역해 팀원의 불확실성을 줄이는 역할을 맡았다.
- **전사 프로젝트 사전 조율**: 기획·인프라·다른 백엔드 팀과의 협업이 필요한 큰 프로젝트는 기술 의사결정 근거를 사전 문서화하고, 본인이 사전 협의를 거쳐 팀 단위 진입 시점을 잡았다. 팀 안으로 들어오는 작업의 우선순위·범위·일정이 명확해진 상태로 시작하도록 했다.

#### 프로젝트 — 자동화 매장 상품 증분 ES 색인 (2023.04)

## 7. 자기소개서 / 지원 동기

### 7.1 쿠팡페이 백엔드 개발 팀장 포지션과의 적합성

21 년차 시니어 백엔드로서 본인이 가장 무게를 두는 자리는 **팀 운영과 시스템 책임을 병행하는 영역** 입니다. 쿠팡페이의 본 포지션을 한 명의 개인 기여자가 아닌 백엔드 팀 단위의 운영·아키텍처 책임자 자리로 이해하고 있고, 이 부분이 본인이 지난 5 년간 (티몬·케이타운포유·리드플레닛) 꾸준히 일해 온 영역과 정확히 맞물립니다.

쿠팡페이의 결제 플랫폼·정산 영역에 본인 경험에서 직접 맞물리는 지점은 세 가지입니다. 첫째, **외부 결제·금융 시스템 연동의 정합성 설계 경험** — MAXVISOR 에서 POS 3 사 (KICC / 스마트로 / 메타시티) 매출 데이터 연동과 키움증권 가상계좌 연동을 직접 설계·구현하면서 외부 시스템 응답 지연·중복·실패 시나리오와 보상 처리를 처음부터 고려한 패턴을 다뤘습니다. 둘째, **매출·정산 데이터의 대사·정합성 보정** — 케이타운포유 BackOffice 스쿼드에서 이벤트 데이터 등록과 주문 오입력 수정 등 수백만 건 규모의 데이터 정합성 보정을, 티몬에서 상품 DB 약 500 만 건 이관과 SKU 단위 구조 개선을 운영했습니다. 셋째, **메시지 정합성 패턴** — 리드플레닛에서 Outbox Pattern 을 도입해 DB 트랜잭션과 메시지 발행을 동시 commit, 데이터 유실 방지와 재처리·상태 추적이 가능한 구조를 운영 중이고, 이 패턴은 결제·정산 영역의 부분 실패 회피 요구와 같은 결입니다.

리딩 측면에서는, 본 포지션이 요구하는 **2 년 이상의 매니지먼트 + cross-functional 협업 + 동시 다중 program 관리** 의 세 축이 본인이 티몬 (최대 13 명) · 케이타운포유 (BO 스쿼드 4 명) · 리드플레닛 세 곳에서 누적해 온 운영 방식과 그대로 매칭됩니다. 큰 기술 결정 (단일 어플리케이션 + 멀티테넌트 전환, 레이어드 → VSA, 자바 1.8 → 17) 은 분석·계획 문서로 트레이드오프를 명시한 뒤 사전 협의 → 단계 분리 → 단계별 회고 형태로 가져가는 방식을 정착시켰는데, 결제 플랫폼처럼 영향 범위가 큰 영역에서는 이 방식이 그대로 필요할 것으로 봅니다.

대규모 시스템 설계와 위험·복잡 마이그레이션 실적 (자바 1.8 + Spring Boot 1.4.2 → 자바 17 + Spring Boot 2.7.18 운영 무중단 전환, 200 인스턴스 → 단일 멀티테넌트 1 인스턴스, 싸이월드 윈도우 → 리눅스 3.2 PB 컨버전) 은 mission-critical 로드맵 정의와 안정적 실행이 필요한 자리에 직접 적용 가능합니다.

### 7.2 결제 도메인 직접 경험에 대한 솔직한 정리

쿠팡페이 규모의 결제 플랫폼을 운영자 입장에서 직접 다룬 경험은 없습니다. 본인이 다룬 결제·금융 관련 영역은 두 갈래입니다.

첫째는 **인접 연동** — MAXVISOR 에서 POS 3 사 + 키움 가상계좌 같은 외부 결제·금융 시스템과의 연동을 처음부터 설계·구현하면서 정합성·재시도·보상 처리를 다룬 경험이고, 둘째는 **정산성 데이터의 대사·정합성** — 케이타운포유 BO 스쿼드 (수백만 건 규모의 주문·이벤트 데이터 보정) 와 티몬 (상품 DB 500 만 건 이관 + SKU 단위 구조 개선) 에서 다룬 매출·주문 데이터의 정합성 보정 경험입니다. 두 영역 모두 결제 플랫폼이 운영자 입장에서 매일 다뤄야 하는 정합성·보정 작업과 같은 결이고, 실무로 한 사이클 이상씩 돌려 본 영역입니다.

직접 결제 플랫폼 운영 경험이 비어 있는 부분은 빠르게 채울 학습 영역으로 보고 있습니다. 결제 도메인의 핵심 트랜잭션 흐름·정산 사이클·이상 거래 탐지·HA/DR 요구는 인접 경험 (외부 결제 연동 정합성 + 정산 대사) 위에 도메인 학습을 더하면 진입 비용이 낮을 것으로 봅니다.

### 7.3 본인이 부족한 영역에 대한 인식

쿠팡페이의 우대 항목 중 **Spark / Hadoop / Cassandra / Kafka / Airflow 같은 분산 데이터 파이프라인 도구의 본격 운영 경험은 없습니다**. 대규모 정산을 분산 파이프라인으로 짜는 흐름이 핀테크 영역의 트렌드라는 점은 인지하고 있고, 본인의 정산 경험은 Java + Spring Batch 베이스의 DB 대사 작업이라는 점을 정직하게 표기합니다.

다만 인접 기술의 운영 경험 위에 학습 진입 비용은 낮을 것으로 봅니다 — **Kafka** 는 RabbitMQ / SQS / SNS + Outbox 패턴 운영 경험 위에 메시징 모델만 익히면 되고, **Spark / Airflow** 는 Spring Batch 기반 대용량 처리 + Python (FastAPI 실무) + DW 구축 경험 위에 도구·DSL 학습을 더하는 단계, **Cassandra** 는 ES / Redis / CouchBase 의 NoSQL 운영 경험 위에 데이터 모델 패턴을 학습하면 되는 것으로 보고 있습니다. **Kubernetes** 도 동일 — Docker 실무와 개념 학습 위에 운영 깊이를 더하는 단계로 봅니다.

**Kotlin** 은 본격 사용 경험이 제한적이지만 Java 베이스라 학습 곡선이 짧을 것으로 봅니다. 실무 진입은 빠르게 가능할 것으로 판단합니다.
