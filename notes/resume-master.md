#### 종단 책임 — Legacy Office 개선 및 아키텍처 변경 (2025.01 ~ 계속)

본인이 2016 ~ 2019 MAXVISOR 시기에 설계·구현한 ERP SaaS 의 후신이다. 초기 구축이 테넌트별 어플리케이션 분리 구조였고 자바 1.8 + Spring Boot 1.4.2 환경이라 기술 부채가 누적되었는데, 그 결과를 같은 인물이 돌아와 풀어가는 사이클이다. 대상 코드베이스는 maxvisor-api 기준 Java 파일 약 2,000 개, 약 25 만 LOC 규모이다.

본인이 주도한 핵심 결정과 결과:

| 영역 | 변경 전 | 변경 후 | 의미 |
|------|--------|--------|------|
| **인스턴스 아키텍처** | 테넌트별 어플리케이션 분리 (약 200 개 인스턴스) | 단일 어플리케이션 + 멀티테넌트 | 인스턴스 수 1 / 200 로 축소, 운영 단순화 |
| **메모리 점유** | 256 ~ 512 GB 환경의 90 %+ | 128 GB 환경의 10 ~ 15 % | 서버 사양·인스턴스 수 동시 축소로 인프라 비용 큰 폭 절감 |
| **파일 저장소** | 자체 파일 서버 (저장 공간 압박) | S3 이전 | 운영 부담·비용·확장성 동시 개선 |
| **도메인 구조** | 잘못 나눠진 레이어드 | Vertical Slice Architecture 점진 전환 | 도메인별 응집도 향상, 향후 변경 영향 범위 축소 |
| **기술 스택 버전** | 자바 1.8 + Spring Boot 1.4.2 | 자바 17 + Spring Boot 2.7.18 (목표 자바 21 + Boot 3.5+) | 보안·성능·생태계 호환성 확보 |

운영 측면에서는 분석·계획을 프로젝트 내 문서화해 이후 작업자가 이어가도 무리 없는 수준의 인계 가능 상태를 유지하고 있다. 쿼리 복잡도 축소와 가독성 개선, 단위·인수 테스트 코드 추가, 개선 작업과 신규 기능 개발 병행을 지속 중이다. 기술 스택은 MySQL, Java 17, Spring Boot, Claude CLI / Codex / IntelliJ / ChatGPT 이다.

**이 시기에 형성된 역량**: 팀 운영과 시스템 책임 병행, MAXVISOR 후신 종단 책임 (전신 시스템 설계자가 7 ~ 9 년 후 본인 부채를 직접 풀어가는 사이클), 멀티테넌트 ERP SaaS 아키텍처 변경 + 인프라 비용 절감, MSA 분리·Outbox 패턴 등 데이터 정합성·운영 안정성 패턴 도입, AI 코딩 워크플로 회사 단위 표준화 (AGENTS.md), 단기 검증 사이클 (광고 플랫폼·DW 각 1 주) 운영.
